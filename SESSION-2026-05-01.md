## GolfCourseAPI.com — Design Decisions Made This Session

**API basics:**
- Two endpoints: `/v1/search?search_query=` and `/v1/courses/{id}`
- Auth: `Authorization: Key <your-key>` header
- Free tier: 300 sessions — not a real constraint with local-first caching
- Register at https://www.golfcourseapi.com/sign-in

**GPS gap:** API has no search-by-coordinates endpoint. Workaround options:
- Option A (best UX): GPS → reverse geocode (Nominatim/OpenStreetMap) → name search → player confirms top 3 matches
- Option B (simpler): Player types course name in search box, taps match
- **Decision deferred** — build Option B first, add GPS later

**Local cache schema** (`localStorage.courseCache`):
```json
{
  "12345": {
    "id": 12345,
    "club_name": "Kamloops Golf & Country Club",
    "course_name": "Main Course",
    "location": { "latitude": 50.67, "longitude": -120.33 },
    "tees": { "male": [...], "female": [...] },
    "cachedAt": "2026-05-01"
  }
}
```
Each course fetched once, cached permanently. Export/Import JSON as backup against Safari cache wipe.

**27/36-hole variations:** `TeeBox.number_of_holes` in the API schema tells us when a course has more than 18 holes. When detected, show a combination picker before the round:
- 27 holes = 3 loops of 9, pick any 2 (A+B, A+C, B+C)
- 36 holes = 4 loops of 9, pick any 2
- App slices the `holes` array into the right 18, round runs normally

**Agreed build sequence:**
1. Register at golfcourseapi.com → get API key (~10 min, Paul does this)
2. Cache layer in `index.html` — `getCourse(id)`, `searchCourses(query)`, localStorage read/write (~1 hr)
3. Course Select screen — search box, results list, tee picker, shown only on first visit to a new course (~1–2 hrs)
4. Combination picker for 27+ hole courses (~1 hr)
5. Wire CR/SR from selected tee into Net Score / HI math (swap hardcoded values for dynamic lookup)
6. GPS auto-detect as later enhancement
